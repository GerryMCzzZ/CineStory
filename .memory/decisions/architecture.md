## ADR-001: 选择 Spring Boot 作为后端框架

**状态**: ✅ 已接受

**日期**: 2025-03-11

**上下文**:
需要构建一个支持异步任务处理、工作流编排、易于扩展的后端服务。项目需要处理长耗时任务（视频生成），支持并发处理和进度跟踪。

**决策**:
使用 Spring Boot 3.x + Java 17 作为后端框架

**理由**:
| 优势 | 说明 |
|------|------|
| 企业级框架 | 成熟稳定，社区活跃 |
| Spring Batch | 内置批处理框架，适合工作流编排 |
| Spring Data JPA | 简化数据访问，提高开发效率 |
| 异步支持 | `@Async` 注解轻松实现异步处理 |
| 生态完善 | 大量第三方库集成 |
| 易于招聘 | 开发者众多，降低招聘难度 |

**后果**:
- **正面**: 开发效率高，扩展性强，维护成本低
- **负面**: 启动时间较长（约 5-10s），内存占用较高（约 512MB-1GB）

**相关技术**:
- Spring Batch（工作流）
- Spring Data JPA（数据访问）
- Spring WebSocket（实时通信）
- Spring Retry（重试机制）

## ADR-002: 数据库选择 MySQL 而非 PostgreSQL

**状态**: ✅ 已接受

**日期**: 2025-03-11

**上下文**:
需要存储项目、切片、视频生成记录等结构化数据。需要在 MySQL 和 PostgreSQL 之间选择。

**决策**:
使用 MySQL 8.0 作为主数据库

**理由**:
1. **团队熟悉度**: 开发团队对 MySQL 更熟悉
2. **查询需求**: 业务查询相对简单，不需要 PostgreSQL 的高级特性
3. **云服务支持**: 主流云服务对 MySQL 支持更好
4. **工具生态**: MySQL 相关工具更丰富

**后果**:
- **正面**: 部署简单，备份恢复方案成熟
- **负面**: 缺少某些高级特性（如 JSON 索引在旧版本支持有限）

**未来考虑**:
如果业务需要复杂查询或 JSON 操作，可考虑迁移到 PostgreSQL

## ADR-003: 文本切片采用多级策略

**状态**: ✅ 已接受

**日期**: 2025-03-11

**上下文**:
小说需要分割成适合视频生成的片段。分割粒度太大会导致视频过长，太小会破坏叙事完整性。

**决策**:
实现三级切片策略：
1. **一级**: 按章节分割（识别"第X章"）
2. **二级**: 按场景边界分割（对话、地点、时间变化）
3. **三级**: 按句子完整性分割

**理由**:
| 级别 | 目标 | 保持 |
|------|------|------|
| 章节 | 叙事单元完整性 | 章节内故事完整 |
| 场景 | 画面连贯性 | 同一场景连续 |
| 句子 | 语义完整性 | 避免断句 |

**实现细节**:
- 使用正则表达式识别章节标题
- 使用 HanLP 进行中文分句
- 检测对话标记 `「」` 保持对话完整
- 场景转换关键词检测（"此时"、"突然"等）
- 上下文重叠（100 字）保持连贯

**后果**:
- **正面**: 语义完整，上下文连贯
- **负面**: 算法复杂度较高，可能误判场景边界

## ADR-004: 使用 Flyway 进行数据库版本管理

**状态**: ✅ 已接受

**日期**: 2025-03-11

**上下文**:
需要管理数据库schema变更，确保开发、测试、生产环境数据库结构一致。

**决策**:
使用 Flyway 进行数据库迁移

**理由**:
1. **约定优于配置**: 自动执行 `db/migration` 目录下的 SQL 文件
2. **版本控制**: 迁移脚本纳入 Git 版本控制
3. **可追溯**: `flyway_schema_history` 表记录执行历史
4. **团队协作**: 避免手动执行 SQL 导致的不一致

**命名约定**:
```
V{版本号}__{描述}.sql
例如: V1__init_schema.sql
```

## ADR-005: 使用本地文件存储记忆系统

**状态**: ✅ 已接受

**日期**: 2025-03-11

**上下文**:
需要为项目创建一个长期记忆系统，存储目标、进度、决策等信息。

**决策**:
使用 `.memory/` 目录下的 Markdown 文件存储项目记忆

**理由**:
| 优势 | 说明 |
|------|------|
| 版本控制 | 纳入 Git，历史可追溯 |
| 可读性 | Markdown 格式，人眼可读 |
| 简单性 | 无需额外依赖，易于维护 |
| 可移植 | 随项目代码一起迁移 |

**结构**:
```
.memory/
├── goals/      # 目标记忆
├── progress/   # 进度追踪
├── decisions/  # 决策记录
├── reviews/    # 代码审查
├── insights/   # 洞察反思
└── sessions/   # 会话记录
```

## ADR-006: 使用 MinIO 存储视频文件

**状态**: ✅ 已接受

**日期**: 2025-03-11

**上下文**:
需要存储大量视频文件（原始视频、中间片段、最终成品）。

**决策**:
使用 MinIO 对象存储服务

**理由**:
1. **S3 兼容**: AWS S3 API 兼容，迁移方便
2. **自托管**: 可自行部署，成本可控
3. **高性能**: 专为对象存储优化
4. **易集成**: Java SDK 完善

**存储策略**:
- 原始视频: `videos/raw/{projectId}/{sliceId}.mp4`
- 中间片段: `videos/segments/{projectId}/{order}.mp4`
- 最终成品: `videos/final/{projectId}/output.mp4`
