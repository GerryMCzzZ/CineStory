# 需求变更管理指南

> 当项目业务需求发生变化时，如何使用 `.memory/` 系统记录和调整

---

## 快速流程

```
需求变更 → 记录到 decisions/ → 更新 goals/ → 调整 progress/ → 继续开发
```

---

## 步骤 1: 记录需求变更

### 创建新的决策记录

在 `.memory/decisions/requirements.md` 中记录需求变更：

```markdown
# 需求变更记录

## REQ-001: 添加用户认证功能

**提出时间**: 2025-03-11
**变更类型**: 新增功能
**优先级**: P0

**原因**:
需要支持多用户，每个用户独立管理自己的项目和配置

**变更内容**:
- 添加用户注册/登录功能
- 添加用户权限管理
- 项目数据按用户隔离

**影响范围**:
- 后端: 添加 UserController、UserService
- 前端: 添加登录/注册页面
- 数据库: 添加 users 表

**状态**: ✅ 已确认

---

## REQ-002: 支持批量项目生成

**提出时间**: 2025-03-11
**变更类型**: 功能增强
**优先级**: P1

**原因**:
用户反馈需要一次处理多个小说

**变更内容**:
- 支持批量上传小说文件
- 支持批量启动生成任务
- 添加任务队列管理

**影响范围**:
- 后端: TaskService 增强
- 前端: 批量操作界面

**状态**: ⏳ 待评估

---
```

## 步骤 2: 更新项目目标

如果需求变更影响项目愿景，更新 `.memory/goals/project_goal.md`：

```markdown
## 项目目标更新记录

### 2025-03-11: 添加用户系统

**变更前**: 单用户工具
**变更后**: 多用户 SaaS 平台

**新增目标**:
- [ ] 支持用户注册和登录
- [ ] 支持多租户数据隔离
- [ ] 添加用户配额管理

```

## 步骤 4: 记录架构调整

如果需求变更影响架构，更新 `.memory/decisions/architecture.md`：

```markdown

## ADR-007: 添加用户认证系统

**状态**: ✅ 已接受

**上下文**:
需要支持多用户，每个用户独立管理项目

**决策**:
- 使用 Spring Security + JWT 实现认证
- 使用 Filter 实现权限校验
- 用户数据通过 @OneToMany 关联

**理由**:
- Spring Security 是成熟方案
- JWT 无状态，适合前后端分离
- 数据隔离在数据库层面实现

**后果**:
- 需要添加用户表和权限表
- 现有 API 需要添加认证检查
- 前端需要添加登录状态管理

---
```

## 给 AI 的需求变更提示词模板

当你需要告诉 AI 新的需求时，可以使用以下模板：

```
## 需求变更通知

【变更类型】
新增功能 / 功能修改 / 功能删除 / 优先级调整

【变更描述】
详细描述需求变更的内容

【变更原因】
为什么需要这个变更

【影响范围】
你认为这个变更会影响哪些模块

【期望结果】
你希望达到什么效果

请帮我：
1. 更新 .memory/decisions/requirements.md
2. 更新 .memory/goals/project_goal.md (如需要)
3. 更新 .memory/progress/pending.md
4. 评估这个变更对现有代码的影响
5. 给出实施建议
```

## 需求变更示例

### 示例 1: 添加新功能

```
## 需求变更通知

【变更类型】新增功能

【变更描述】
需要添加"配音生成"功能，为生成的视频自动配音

【变更原因】
纯视频不够生动，需要配音增强表现力

【影响范围】
- 后端: 新增 TTS 服务模块
- 前端: 配音配置界面
- 工作流: 视频生成后添加配音步骤

【期望结果】
用户可以自动为视频添加AI配音
```

### 示例 2: 调整功能优先级

```
## 需求变更通知

【变更类型】优先级调整

【变更描述】
将"自定义风格"功能从 P2 提升到 P0

【变更原因】
用户反馈对自定义风格需求强烈

【影响范围】
- 开发顺序调整
- 可能需要提前完成风格模板 CRUD

【期望结果】
用户可以更快使用自定义风格功能
```

### 示例 3: 删除功能

```
## 需求变更通知

【变更类型】功能删除

【变更描述】
暂时取消"社区功能"（作品分享、模板市场）

【变更原因】
资源有限，优先核心功能

【影响范围】
- 删除 community 相关模块
- 简化前端页面

【期望结果】
聚焦核心功能，加快 MVP 上线
```

## 需求变更影响评估

AI 会自动评估：

| 评估项 | 说明 |
|--------|------|
| **代码影响** | 哪些文件需要修改/新增/删除 |
| **架构影响** | 是否需要调整系统架构 |
| **进度影响** | 对当前开发进度的影响 |
| **成本影响** | 预计增加多少开发工作量 |
| **风险评估** | 可能引入哪些新的风险 |

## 最佳实践

### ✅ 推荐做法

1. **大变更先讨论**: 重大需求变更先记录到 `pending.md`，确认后再实施
2. **保持追溯**: 所有变更都要有记录，包括被拒绝的需求
3. **更新决策**: 每个重要决策都要记录 ADR
4. **定期回顾**: 每周回顾一次需求变更，评估是否合理

### ❌ 避免做法

1. **口头约定**: 不要只口头约定，要写下来
2. **直接修改代码**: 需求变更要先记录再实现
3. **删除历史**: 不要删除或修改旧的变更记录
4. **忽略影响**: 不要忽视变更对现有功能的影响
